# Semicolon; 팀 소개 페이지 협업 실습

본 레포지토리는 인공지능 소프트웨어학과 '세미콜론(Semicolon)' 동아리 팀원들의 Git 및 GitHub 협업 프로세스 실습을 위해 생성되었습니다.

## 📌 프로젝트 개요
* **목적:** 브랜치(Branch) 분기, 커밋(Commit), 푸시(Push), 풀 리퀘스트(PR), 병합(Merge) 및 충돌(Conflict) 해결 등 실제 실무 협업 흐름의 숙달.
* **참여 인원:** 지도자(팀장) 1명, 팀원 5명

## 📁 디렉토리 구조
```text
📦 SEMICOLON-GIT-PRACTICE
 ┣ 📂 css/            # 모든 스타일시트 파일 (.css)
 ┃ ┣ 📜 index.css     # 메인 및 공통 레이아웃 스타일
 ┃ ┗ 📜 [이름].css    # 팀원 개별 프로필 전용 스타일
 ┣ 📂 img/            # 공통 및 개인 프로필 이미지 에셋
 ┣ 📂 pages/          # 팀원 개별 프로필 HTML 페이지
 ┃ ┗ 📜 [이름].html   # 팀원 개별 프로필 파일
 ┣ 📜 index.html      # 팀 소개 메인 페이지
 ┗ 📜 README.md       # 프로젝트 가이드 및 컨벤션
 ```

## 🛠 코딩 컨벤션 (Coding Convention)
동아리원들은 다음 규칙을 반드시 준수하여 작업해야 합니다.

### 1. 파일 관리 및 구조
- **분리 원칙**: 모든 스타일은 외부 CSS 파일로 작성하며, HTML 내부의 style 태그 사용은 금지합니다.
- **파일명 일치**: 개별 프로필 작업 시 HTML 파일명과 CSS 파일명을 동일하게 설정합니다.
> 예: pages/suyong.html 작업을 위한 스타일은 css/suyong.css에 작성
- **이미지 저장**: 모든 이미지는 반드시 img/ 폴더 안에 추가한 뒤 상대 경로를 사용합니다.

### 2. Git 협업 규칙 (Github Flow)
원활한 협업과 프로젝트 관리를 위해 아래의 브랜치 전략과 커밋 컨벤션을 엄격히 준수합니다.

#### 🌿 브랜치(Branch) 전략
| 브랜치명 | 용도 및 설명 |
| :--- | :--- |
| `main` | 최종 배포 및 병합용 (**개인적인 직접 푸시 절대 금지**) |
| `feature/이름` | 새로운 기능 추가 및 개발 |

#### 📝 커밋 메시지(Commit Message) 컨벤션
커밋 메시지는 반드시 `타입: 제목` 형식으로 작성합니다. (예: `feat: 로그인 페이지 마크업 추가`)

| 타입 (Type) | 설명 |
| :--- | :--- |
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `design` | 디자인 변경 |
| `docs` | 문서 수정 (README.md 등) |
| `refactor` | 코드 리팩토링 |
| `chore` | 단순 작업 (파일 이동, 삭제, 포맷팅 등) |

#### 📌 이슈 및 PR (Issue & Pull Request)
* **이슈 (Issue)**: 선택사항으로 작업 전 해결해야 할 과제나 추가할 기능을 먼저 이슈로 등록하여 팀원들과 작업 내역을 공유합니다.
* **PR (Pull Request)**: 필수 사항으로 작업 완료 후 개인 브랜치에서 `main` 브랜치로 병합(Merge) 요청을 보냅니다.
  * PR 제목은 커밋 메시지와 동일한 컨벤션을 따릅니다.
  * PR 본문에는 작업 내용과 변경 사항을 상세히 기술합니다..
  * Issue 가 있는 경우 반드시 Issue와 매핑하여 PR 본문 맨 마지막에 Close #이슈번호 를 작성해주세요.
